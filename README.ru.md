SSH-терминал в браузере. Обычный HTTP, без сборки, без отдельных сервисов.

- 📦 Без npm, без pip — кладёшь файлы и запускаешь
- 🌐 Корпоративные сети, где открыт только HTTPS: работает без WebSocket
- ⭐ Сессия живёт после закрытия вкладки, перезагрузки и рестарта бэкенда — до 72 ч (через tmux на удалённом хосте)

```
┌─ Your browser ─┐    HTTPS     ┌── websh host ──┐     SSH      ┌──── Remote ────┐
│                │              │                │              │                │
│    xterm.js    │─── POST ────►│   server.py    │◄────────────►│      bash      │
│                │◄─── SSE ─────│    (Python)    │              │     + tmux     │
│                │              │                │              │                │
└────────────────┘              └────────────────┘              └────────────────┘
```

## Как это работает

Три части:

- **Браузер.** xterm.js рисует терминал. Каждое нажатие летит POST-ом на `/api/input`.
- **websh-хост.** `server.py` держит каждое SSH-подключение как PTY-подпроцесс и стримит вывод обратно через Server-Sent Events на `/api/stream`. Тот же процесс отдаёт фронтенд — отдельный веб-сервер не нужен.
- **Удалённый хост.** Куда ты заходишь по SSH. По желанию оборачивается в tmux, чтобы сессия пережила переподключение.

Если прокси буферизирует SSE (бывает на shared-хостинге), клиент переключается на long-polling по `/api/output` для этой сессии. Медленнее, но работает.

Shared-хостинг не даёт держать долгоживущий процесс? Положи `api.php` рядом с `server.py`. PHP-shim запустит бэкенд при первом запросе и будет проксировать API-вызовы к нему.

**Почему не WebSocket?** Многие shared-хостинги его не проксируют — websh должен работать и там. SSE даёт ту же низкую задержку поверх обычного HTTP и проходит через любой HTTPS-прокси без апгрейда протокола.

Глубже про устройство — детектор буферизации, защита от потери байт при дисконнекте, локальное эхо-предсказание, ожидание через selectors — см. [`docs/sse-transport.md`](docs/sse-transport.md).

## Требования

- **Бэкенд.** Python 3.5+ с `ssh` в PATH. Только stdlib — никаких pip-зависимостей.
- **Браузер.** Любой современный. xterm.js грузится с CDN.
- **Опциональный proxy на shared-хостинге.** PHP 5.3+ с расширением `curl`.
- **Опциональный reverse-proxy.** nginx, Caddy или Apache.

## Особенности

### 🖥️ Терминал

Настоящий xterm.js — копирование выделением, вставка правой кнопкой, поиск по скроллбэку (`Ctrl+Shift+F`), масштаб (`Ctrl+±`), полноэкранный режим (`F11`).

- Разделение на панели, горизонтально и вертикально, с перетаскиваемыми разделителями
- Переключение панелей: `Ctrl+Tab` / `Ctrl+Shift+Tab`
- Тёмная и светлая темы (выбор сохраняется)
- Выбор шрифта (⚙) с живым предпросмотром — JetBrains Mono, Fira Code, IBM Plex Mono, Roboto Mono, Source Code Pro, Inconsolata или системный. Кастомный размер, межстрочный, начертание

### 🔁 Постоянные сессии

Поставь галочку **Persistent session** при подключении — websh обернёт shell в tmux-сессию на удалённом хосте. Закрой вкладку, перезагрузи компьютер, перезапусти `server.py`: панель переподключится к той же tmux-сессии со скроллбэком и работающими процессами. См. [Постоянные сессии (tmux)](#постоянные-сессии-tmux).

- Кнопка реконнекта при разрыве; красный баннер при auth-fail
- URL-якоря (`#connect=Production`) для прямых ссылок и закладок
- Сохранённые подключения в `localStorage` браузера

### 📁 Передача файлов

Загрузка и скачивание без `scp`.

- **Загрузка.** Выбираешь файлы; браузер стримит байты через пиггибэк-канал SSH ControlMaster (`cat > $HOME/<tmp>` без PTY, без base64, один POST на файл). На постоянных (tmux) панелях файл попадает в `pane_current_path` автоматически — vim/less/htop в фоне не дёргаются. Не-постоянные панели набирают `mv` в активный шелл с защитой от alt-screen. Авто-инкремент при коллизии имён. Прогресс через xhr.upload, очередь, отмена в полёте.
- **Скачивание.** Выделяешь имя файла в терминале, нажимаешь Download.
- **Экспорт скроллбэка.** Сохраняет текущий буфер в текстовый файл. На постоянных панелях берёт реальный tmux-скроллбэк через `tmux capture-pane`.

### 🔐 Профили подключений

От свободного «введи хост и поехали» до строго ограниченного click-to-connect.

- Аутентификация по паролю и SSH-ключу
- Серверные профили в `websh.json` — креды остаются на сервере, браузер их не видит
- Два типа: **Ready** (сохранённые креды) и **Prompt** (целевой хост из allowlist, пользователь сам вводит пароль)
- `allowed_users` / `denied_users` для каждого профиля
- SSH-опции для отдельных профилей (`ProxyJump`, `StrictHostKeyChecking`, …)
- `restrict_hosts: true` полностью прячет свободную форму

### 🚀 Развёртывание

- **Shared-хостинг.** Заливаешь по FTP 4 файла + `assets/`; `api.php` сам стартует бэкенд. SSH на хост не нужен.
- **Только Python.** Бэкенд сам отдаёт фронтенд — ноль зависимостей.
- **Docker, systemd, reverse-proxy.** Рецепты в комплекте.
- Транспорт по обычному HTTP с автоматическим fallback на long-poll для хостов, которые буферизируют SSE.

## Сценарии

- **Корпоративный фаервол.** SSH-порт закрыт, открыт только HTTPS. websh туннелит через стандартный HTTPS.
- **Нет родного терминала.** Chromebook, iPad, kiosk. Любой браузер становится терминалом.
- **Доступ для клиента.** Даёшь клиенту ссылку на его собственный сервер. URL-якоря (`#connect=ServerName`) для прямых ссылок.
- **UI на bastion.** Ставишь websh на jump-host, ходишь во внутренние серверы из любого браузера.
- **Аварийный доступ с чужой машины.** Открыл URL — ты внутри.
- **Воркшопы.** Студентам не нужно ничего ставить локально.

## Быстрый старт (на своей машине)

```bash
git clone https://github.com/dolonet/websh.git
cd websh
python3 server.py
```

Открой http://localhost:8765 — всё. Без pip install, без npm, без сборки.

Нужны Python 3.5+ и `ssh` в PATH. По умолчанию сервер слушает
`127.0.0.1`; задай `HOST=0.0.0.0`, чтобы выставить его в локальную сеть.

## Быстрый старт (shared-хостинг)

**SSH-доступ не требуется.** Заливаешь файлы по FTP, открываешь в браузере.

Типичная структура каталогов на shared-хостинге:

```
/home/user/
  example.com/              <- корень сайта
    websh.json              <- конфиг (ВНЕ www — недоступен по HTTP)
    www/                    <- веб-корень (публичный)
      console/
        index.html          <- фронтенд
        websh.js            <- логика фронтенда
        api.php             <- PHP-прокси
        server.py           <- бэкенд (автозапускается api.php)
        assets/             <- бренд-SVG (логотипы, светлая/тёмная)
```

**Шаги:**

1. Создай папку в веб-корне (например `www/console/`)
2. Залей туда `index.html`, `websh.js`, `api.php`, `server.py` и папку `assets/`
3. Открой `https://your-host/console/` в браузере

Всё. `api.php` сам запускает `server.py` при первом запросе.

> **Про путь к конфигу:** `api.php` ищет `websh.json` на два каталога
> выше себя (т. е. в корне сайта, над `www/`). Это работает у большинства
> хостинг-провайдеров. Если у тебя другая раскладка — задай переменную
> окружения `WEBSH_CONFIG` или поправь путь в `api.php` строка 34.

### Решение проблем

**«Backend unavailable» или белая страница:**
- Проверь, что Python 3 установлен: `python3 --version`
- Проверь, что `ssh` доступен: `which ssh`
- Некоторые shared-хостинги отключают `exec()` в PHP — спроси хостера или посмотри `phpinfo()`

**Конфиг не подхватывается:**
- Проверь путь к `websh.json` — `api.php` ищет на два каталога выше по умолчанию
- Если раскладка отличается — задай `WEBSH_CONFIG=/full/path/to/websh.json`
- Проверь синтаксис JSON: `python3 -c "import json; json.load(open('websh.json'))"`

**Порт занят:**
- Возможно, уже запущен другой `server.py`: `ps aux | grep server.py`
- Поменяй порт: `PORT=8766 python3 server.py`

## Серверные подключения (опционально)

Преднастроенные подключения, чтобы пользователи просто кликали для
коннекта — без паролей на клиенте. Создай `websh.json` в **корне
сайта** (не в `www/`):

```json
{
  "restrict_hosts": false,
  "connections": [
    {
      "name": "Production",
      "host": "server.example.com",
      "port": 22,
      "username": "deploy",
      "password": "secret"
    }
  ]
}
```

Полный пример с аутентификацией по SSH-ключу и кастомными SSH-опциями см. в `websh.json.example`.

> **В этом файле — пароли, держи его вне веб-корня.**
> Он не должен быть доступен по HTTP. Если раскладка хостинга не
> совпадает с диаграммой выше — задай `WEBSH_CONFIG`.

### SSH-опции для отдельных подключений

Переопредели поведение SSH для конкретных подключений:

```json
{
  "name": "Strict server",
  "host": "secure.example.com",
  "username": "admin",
  "password": "secret",
  "ssh_options": {
    "StrictHostKeyChecking": "yes",
    "ProxyJump": "bastion.example.com"
  }
}
```

### Типы подключений: Ready и Prompt

Каждая запись `connections[]` относится к одному из двух типов —
определяется автоматически по наличию `password` или `key`:

- **Ready** — креды (`password` или `key`) хранятся на сервере.
  Пользователь кликает по карточке и подключается. Браузер кредов
  никогда не видит.
- **Prompt** — нет ни `password`, ни `key`. Запись работает как
  целевой хост из allowlist: пользователь кликает по карточке,
  открывается ручная форма с предзаполненными полями (host/port
  заблокированы, username заблокирован, если он фиксирован), и
  пользователь сам вводит пароль или ключ.

Записи типа Prompt могут опционально нести `allowed_users` (whitelist)
или `denied_users` (blacklist), чтобы ограничить, какие имена
пользователей могут подключаться. `allowed_users` побеждает, если оба
заданы. Эти правила игнорируются, если у записи фиксированный
`username` (выбора всё равно нет). Сохранение введённых кредов локально
через галочку «Save this connection» работает так же, как из свободной
ручной формы.

```json
{
  "name": "Shared DB",
  "host": "db.example.com",
  "port": 2222,
  "allowed_users": ["alice", "bob"]
}
```

### Режим ограничения (restrict)

Установи `"restrict_hosts": true`, чтобы полностью спрятать свободную
ручную форму подключения. Пользователи смогут идти только через
сконфигурированные карточки. Прямые POST-запросы по ручному пути на
`/api/connect` (минуя UI) тоже отвергаются. Если подключение всего
одно, UI его автоматически выбирает при загрузке: Ready подключается
сразу, Prompt показывает заблокированную форму, готовую к вводу пароля.

### Замечание о безопасности списков пользователей

`allowed_users` / `denied_users` действуют только внутри **именованного**
flow подключения (`{connection: "<name>"}` на `/api/connect`). Когда
`restrict_hosts` выключен, свободная ручная форма и прямые POST'ы по
ручному пути этими списками не ограничены — это allowlist для UX-а
твоей команды, а не граница защиты от настойчивого вызывающего.
Сочетай с `restrict_hosts: true`, если правила должны действовать и
против прямого API-доступа.

### URL-якоря

Прямая ссылка на серверное подключение:

```
https://your-host/console/#connect=Production
```

Подключение происходит автоматически при загрузке страницы — удобно
для закладок и ссылок поддержки.

## Постоянные сессии (tmux)

Поставь галочку **Persistent session** в форме подключения — и
удалённый шелл оборачивается в tmux-сессию на целевом хосте
(`tmux new-session -A -D -s websh-<slot>`). Закрой вкладку, перезагрузи
страницу или перезапусти `server.py` — панель переподключится к той же
сессии со scrollback'ом и работающими процессами.

**Требования.** На целевом хосте должен быть установлен `tmux` (любой
свежий). Если его нет — flow подключения покажет попап с предложением
fallback'а на короткоживущую (не-постоянную) сессию.

**Как работает реаттач.** Каждая постоянная панель хранит свой slot id
в `localStorage` рядом с записью подключения. На refresh фронтенд
заново открывает панель с тем же slot id, и tmux переподключает к
существующей сессии. Slot id живёт на одну инстанцию панели — закрытие
панели через `[x]` slot для повторного использования не освобождает.

**Завершение сессии.** Клик по `[x]` на постоянной панели открывает
модалку (Cancel / Terminate session / Terminate and never ask again).
«Terminate» отсылает на цель `tmux kill-session` перед закрытием
панели. Если просто закрыть вкладку без terminate — сессия на цели
остаётся живой, и можно переподключиться позже.

**Watchdog по idle-TTL.** При создании сессии рядом с шеллом
порождается detached POSIX-sh watchdog. Он опрашивает tmux и убивает
сессию, как только она проболталась без аттача `WEBSH_TMUX_IDLE_TTL`
секунд (по умолчанию 72 ч; `0` отключает). Watchdog переродится к init
через `nohup` и переживает рестарты `server.py`. Активные (с
аттачем) сессии обновляют таймер на каждом polling'е, так что
долгоиграющая работа не пожинается просто из-за короткого
дисконнекта.

**Per-connect tmux-опции.** Каждый постоянный коннект выполняет
`tmux new-session … \; set -g …`, так что небольшой набор tmux-опций
применяется единообразно независимо от того, что прописано на целевом
хосте. Mouse-режим вшит без права отключения; два тогла в панели
Options остаются настраиваемыми и пушатся в работающие панели в
момент изменения, так что новое поведение применяется без реконнекта:

- **Mouse** (всегда включён, тогла нет) — `set -g mouse on`. Колесо
  прокручивает scrollback tmux в шелле; alt-screen приложения (vim,
  less, htop) получают сырые mouse-события. Зажми Shift, чтобы обойти
  tmux-выделение и использовать нативное выделение текста браузером.
- **Auto-copy** (тогл) — `set-clipboard on`. Выделения в copy-mode
  tmux пушатся в системный буфер обмена через OSC52 (xterm.js их
  пропускает).
- **Scrollback** (число) — `history-limit` (по умолчанию 100 000).
  Сколько строк на панель tmux хранит.

Сервер принимает настраиваемые параметры только по фиксированному
allow-list (`set-clipboard`, `history-limit` зажат в 100..10 М); всё
остальное — включая устаревшее поле `tmux_mouse` от старых клиентов —
молча отбрасывается, так что устаревший или враждебный клиент не
сможет подсунуть лишние `set -g` строки. Mouse остаётся включён в
любом случае.

`set -g` строки выполняются после собственного `~/.tmux.conf` целевого
хоста и переопределяют там совпадающие опции.

## Настройка

Переменные окружения для `server.py`:

| Переменная | По умолчанию | Описание |
|---|---|---|
| `PORT` | `8765` | Порт для прослушивания |
| `HOST` | `127.0.0.1` | Адрес для bind |
| `SESSION_TIMEOUT` | `300` | Таймаут простоя в секундах |
| `MAX_SESSIONS` | `50` | Максимум одновременных SSH-сессий |
| `WEBSH_CONFIG` | *(автодетект)* | Путь к файлу конфига `websh.json` |
| `TRUSTED_PROXIES` | `127.0.0.1` | IP-адреса через запятую, которым доверяем `X-Forwarded-For` |
| `MAX_BG_SESSIONS` | `50` | Максимум фоновых SSH-сессий (загрузка/скачивание файлов) |
| `RATE_LIMIT_MAX` | `50` | Максимум попыток на `/api/connect` с одного IP за окно |
| `RATE_LIMIT_WINDOW` | `60` | Окно rate-limit'а в секундах |
| `WEBSH_TMUX_IDLE_TTL` | `259200` | Сколько секунд detached постоянная tmux-сессия может бездельничать на цели до жатвы (по умолчанию 72 ч, `0` отключает) |
| `WEBSH_TMUX_WATCHDOG_POLL` | `300` | Интервал между проверками idle-TTL watchdog'а на цели в секундах |

PHP-прокси читает `WEBSH_PORT` (по умолчанию `8765`), чтобы найти бэкенд.

## Развёртывание

### Shared-хостинг (PHP + Python)

Залей четыре файла (`index.html`, `websh.js`, `api.php`, `server.py`)
и папку `assets/` (бренд-SVG) в веб-каталог. Бэкенд запустится сам.

Для ручного контроля (например, кастомный путь к конфигу):

```bash
WEBSH_CONFIG=/path/to/websh.json nohup python3 server.py &
```

### Только Python (без PHP)

Бэкенд может сам отдавать фронтенд — без PHP и отдельного веб-сервера:

```bash
HOST=0.0.0.0 python3 server.py
```

Открой `http://your-host:8765/` в браузере. Бэкенд отдаёт статику
(`index.html`, `websh.js`, `assets/*.svg`) из того же каталога, где
лежит `server.py`, и обрабатывает API-запросы на том же порту.
См. [HTTPS через reverse-proxy](#https-через-reverse-proxy) ниже.

### Docker

```bash
docker build -t websh .
docker run -d -p 8765:8765 -e HOST=0.0.0.0 websh
```

Открой `http://localhost:8765/` — бэкенд сам отдаёт фронтенд.
См. [HTTPS через reverse-proxy](#https-через-reverse-proxy) ниже.

### systemd

```bash
# Создаём отдельного пользователя
useradd -r -s /bin/false websh

mkdir -p /opt/websh
cp server.py index.html websh.js /opt/websh/
cp websh.service /etc/systemd/system/
systemctl enable --now websh
```

### HTTPS через reverse-proxy

Поставь nginx или Caddy впереди для TLS-терминирования:

```nginx
server {
    listen 443 ssl;
    server_name ssh.example.com;

    location / {
        proxy_pass http://127.0.0.1:8765;
        proxy_read_timeout 60s;
    }
}
```

`proxy_read_timeout` должен с запасом превышать и окно long-poll
(10 с), и интервал SSE-keepalive (15 с) — 60 с оставляет хороший
запас. Бэкенд выставляет `X-Accel-Buffering: no` на SSE-ответ, так
что nginx флашит каждое событие сразу без дополнительной настройки.
Если прокси крутится на другом хосте — добавь его IP в
`TRUSTED_PROXIES`, чтобы rate-limit считал по реальному IP клиента —
см. [Rate-limit и прокси](#rate-limit-и-прокси).

## Аутентификация и безопасность

**websh намеренно не включает собственный слой аутентификации.**
Он задуман лёгким — добавляй контроль доступа на уровне веб-сервера:

- **Apache:** `.htaccess` с `AuthType Basic` + `AuthUserFile`
- **nginx:** директива `auth_basic`
- **Cloudflare Access**, **Tailscale Funnel** или похожие zero-trust инструменты
- IP allowlisting через правила фаервола

### SSH host keys

По умолчанию бэкенд подключается с `StrictHostKeyChecking=no`, чтобы
избежать интерактивных prompt'ов. **Это делает первое подключение к
любому хосту уязвимым к MITM** — серверная идентичность не проверяется.

Это приемлемо когда:
- Ты подключаешься к своим серверам в доверенной сети
- Соединение идёт через зашифрованный туннель (VPN, Tailscale и т. п.)

Чтобы включить проверку host-key для конкретных подключений — добавь
`ssh_options` в `websh.json`:

```json
"ssh_options": {"StrictHostKeyChecking": "yes"}
```

### Сохранённые подключения и пароли

Сохранённые подключения в браузере хранятся в `localStorage`
**в открытом виде**, включая пароли. Любой JavaScript на том же origin
(включая XSS-уязвимости) сможет их прочитать.

Если для твоего сценария это неприемлемо:
- Используй серверные подключения (`websh.json`) — пароли остаются на сервере, до браузера не доезжают
- Не сохраняй подключения в браузере — используй SSH-ключи
- Ограничь доступ к URL websh доверенными сетями

### Rate-limit и прокси

Попытки подключения по умолчанию ограничены **50 на один IP в минуту**
(настраивается через `RATE_LIMIT_MAX` и `RATE_LIMIT_WINDOW`). IP клиента
определяется из `X-Forwarded-For` **только** когда запрос приходит с
IP, перечисленного в `TRUSTED_PROXIES` (по умолчанию: `127.0.0.1`).
Прямые подключения всегда используют TCP peer-адрес —
`X-Forwarded-For` подделать нельзя.

Если твой reverse-proxy крутится на другом хосте — добавь его IP:

```bash
TRUSTED_PROXIES=127.0.0.1,10.0.0.5 python3 server.py
```

### Валидация ввода

- Значения host и username, начинающиеся с `-`, отвергаются (защита от инжекта SSH-флагов)
- Session ID валидируются как UUID
- Размеры терминала зажимаются в безопасный диапазон
- `MAX_SESSIONS` ограничивает одновременные пользовательские сессии; `MAX_BG_SESSIONS` отдельно ограничивает сессии передачи файлов
